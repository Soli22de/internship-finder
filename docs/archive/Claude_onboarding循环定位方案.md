# Claude AI 卡在 Onboarding 循环问题定位方案

## 目标
- 精确定位“登录后总是跳回 onboarding 页面”的根因。
- 给出可验证的修复路径，避免继续盲目尝试。
- 输出可复现证据，确保后续与官方支持沟通时一次性说明白。

## 问题画像与工作假设
- 现象：用户已尝试多种方法，但每次最终仍重定向到 onboarding。
- 初步判断：这是“前端路由结果”，其上游通常来自会话、账户状态、组织/权限、网络响应异常、浏览器存储污染中的一类。
- 关键原则：逐层排除，不做重复试错；每一步都产出“证据 + 结论”。

## 执行步骤

### 第 1 阶段：建立可复现基线
1. 固定单一测试环境（浏览器、网络、账号、设备），只改一个变量。
2. 记录完整时间线：点击登录、回调完成、首次进入页面、被重定向到 onboarding 的时间点。
3. 确认是否“100% 必现”或“间歇复现”，并标注触发条件（刷新、切换 tab、重新登录等）。

### 第 2 阶段：抓取重定向链与关键接口证据
1. 在浏览器开发者工具抓取一次完整登录到跳转过程的 Network 记录（含 Preserve log）。
2. 提取关键证据：
   - 首次业务页面请求是否返回 30x 到 onboarding。
   - 用户信息/会话相关接口状态码（200/401/403/5xx）。
   - 响应体中的状态字段（是否显示未完成 onboarding、无组织、权限不足、会话失效）。
3. 对比 Cookie 与 Storage 变化（登录前后 token/session 是否写入成功、是否被立即清除或覆盖）。

### 第 3 阶段：账户与租户状态核验
1. 核验账号是否处于“待完成引导”状态（可能由上次中断、企业空间策略、年龄/地区合规流程触发）。
2. 检查是否绑定了多个 workspace/organization，且默认空间无有效权限导致回落 onboarding。
3. 验证登录方式是否混用（同邮箱不同 IdP，如 Google 与邮箱密码），导致后端识别为不同身份档案。

### 第 4 阶段：浏览器与客户端侧污染排查
1. 以“全新浏览器 Profile / 隐私窗口 / 无扩展”进行对照实验，验证是否是本地状态污染。
2. 定位干扰源：
   - 隐私插件、脚本拦截器、广告拦截、企业安全插件。
   - 第三方 Cookie 策略、SameSite/跨域限制、时间不同步导致 token 立即过期。
3. 若对照环境正常，回溯到原环境逐项恢复，定位具体冲突项。

### 第 5 阶段：网络与地区策略排查
1. 对比不同网络出口（家庭网/手机热点）与代理开关状态，确认是否存在风控或边缘节点异常。
2. 检查 DNS、TLS 中间人代理、企业网关注入导致的请求改写/阻断。
3. 若某网络稳定复现而另一网络正常，判定为网络路径或策略问题并给出绕行方案。

### 第 6 阶段：根因归类与修复方案落地
1. 根据证据将根因归类为以下之一：
   - 账户状态异常；
   - 组织/权限映射异常；
   - 会话写入或续期失败；
   - 本地浏览器状态污染；
   - 网络/安全策略拦截。
2. 为每类根因给出对应修复动作与回归验证步骤。
3. 输出“最小闭环验证”：登录后连续 3 次刷新不再跳回 onboarding，跨会话仍保持正常。

### 第 7 阶段：升级支持工单（若需）
1. 汇总可提交证据包：时间戳、请求 ID、关键接口返回、账号标识脱敏信息、复现步骤。
2. 形成结构化工单模板，确保官方支持可直接定位后端状态或策略问题。

## 交付物
- 一份“问题定位报告”：包含根因、证据、修复动作、验证结果。
- 一份“应急绕行建议”：在根因修复前可用的临时访问方案。
- 一份“支持工单模板”：用于与官方支持高效沟通。

## 验收标准
- 已明确唯一主根因或给出最多两个候选根因并附排他证据。
- 修复后在目标环境不再出现 onboarding 循环。
- 提供可复现与可验证的完整记录，后续可审计。
